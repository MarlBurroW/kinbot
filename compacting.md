# KinBot — Compacting algorithm

Compacting maintains a synthesized **working memory** for each Kin. It summarizes older exchanges so that the context sent to the LLM stays within the token window limits, while preserving important information.

> **Language convention**: Compacting and extraction prompts are written in English as the base language. The summary is generated in English regardless of the conversation language, since it is internal context for the LLM (not shown to users).

---

## Progressive context compaction pipeline

Before compacting (DB-level summarization) even runs, KinBot applies a **3-zone progressive compaction pipeline** to the in-memory context sent to the LLM. This reduces token usage without any LLM calls:

### Zone 1 — Intact (most recent)

The last N tool call groups are kept **fully intact** with complete tool results. Controlled by `TOOL_RESULT_MASK_KEEP_LAST` (default: 2).

### Zone 2 — Observation/Truncated (middle)

Older tool results are **truncated** to a maximum character limit. Long assistant and user messages in this zone are also trimmed. Controlled by:
- `OBSERVATION_COMPACTION_WINDOW` (default: 10) — number of turns in this zone
- `OBSERVATION_MAX_CHARS` (default: 200) — max characters for truncated tool results

### Zone 3 — Collapsed (oldest)

The oldest tool call groups are **fully collapsed** to one-line summaries (e.g. `[Tool result collapsed — 3.2k chars saved]`). This aggressively saves tokens on historical tool interactions.

### Emergency token budget

`HISTORY_TOKEN_BUDGET` (default: 0 = disabled) acts as a last-resort safety net. If set, messages are trimmed from the oldest end when the total estimated tokens exceed this budget. This should rarely activate since the pipeline + compacting handle context size.

---

## When compacting triggers

Compacting is evaluated **after each LLM turn** (after the Kin has responded). It uses a **turn-count-based** trigger:

| Condition | Threshold (configurable) |
|---|---|
| Number of non-compacted turns | > `compacting.batchTurns` + `compacting.minKeepTurns` (default: 10 + 15 = 25) |

A **turn** is defined as one user message plus all following messages (assistant responses, tool calls, tool results) until the next user message. This avoids false triggers in tool-heavy conversations where a single turn can produce 10-30 messages.

> **Important**: messages with `redact_pending = 1` are **excluded** from compacting. Compacting waits until redaction is effective before including them.

### Per-Kin configuration

Each Kin can override the global compacting settings via its `compactingConfig` (stored as JSON in `kins.compacting_config`). Available per-Kin fields:

| Field | Type | Description |
|---|---|---|
| `turnThreshold` | number | Total turn threshold before compaction triggers. Overrides `batchTurns + minKeepTurns`. A 40/60 split is derived automatically (40% batch, 60% keep) |
| `compactingModel` | string | Model for compaction LLM calls. Overrides `COMPACTING_MODEL` |
| `compactingProviderId` | string | Provider ID for the compacting model |

Per-Kin config is managed in the UI via the **Compaction** settings tab.

## Incremental micro-batch compacting

Instead of compacting all messages at once, KinBot uses **incremental batching**: messages are summarized in fixed-size micro-batches (by turn count) from the oldest end, spreading LLM cost over many small calls instead of one expensive spike.

### Algorithm

#### Step 1 — Select a batch

```
non_compacted = messages after last active snapshot
                WHERE redact_pending = 0
                ORDER BY created_at ASC

turn_count = count of user messages in non_compacted

// Only proceed if enough turns have accumulated
if (turn_count <= turnThreshold) → skip

// Find the cut point: after batchTurns user messages
cutIndex = index of the (batchTurns + 1)th user message
batch = non_compacted[0 .. cutIndex]

// Verify enough turns remain as raw context
remainingTurns = user messages after cutIndex
if (remainingTurns < minKeepTurns) → skip
```

- `batchTurns` (default: 10) — number of turns per micro-batch
- `minKeepTurns` (default: 15) — minimum non-compacted turns to keep as raw context

#### Step 2 — Build the compacting prompt

The summary is generated by a dedicated LLM call (model configurable via `compacting.model`, otherwise the Kin's model).

Verbose tool results (>500 chars) in the batch are **masked** before being sent to the summarization LLM — replaced with `[Tool result — N chars, collapsed for summarization]`. This reduces the token cost of summarization calls and prevents oversized API payloads.

```
System: You are an assistant specialized in conversation summarization.
Your role is to produce a faithful and complete summary of the exchanges below.

Time range: {firstTimestamp} to {lastTimestamp} ({batchSize} messages)

Rules:
- Preserve ALL important facts, decisions made, commitments, and expressed preferences
- Preserve the identity of who said what (use names/pseudonyms)
- Preserve the context of ongoing or recently completed tasks
- Preserve results of research, calculations, or work performed
- Do not invent anything — only summarize what is explicitly present
- Be concise but complete. Prefer bullet points for facts
- If a previous summary exists, integrate it into your new summary

{previous_summary ? "## Previous summary\n\n" + previous_summary : ""}

## Exchanges to summarize

{formatted_messages}
```

#### Step 3 — Generate the summary

```typescript
const summary = await llm.chat({
  model: config.compacting.model ?? kin.model,
  messages: [{ role: 'system', content: compactingPrompt }],
  // No streaming, no tools
})
```

#### Step 4 — Save the snapshot

```typescript
// Create the new snapshot
await db.insert(compactingSnapshots).values({
  id: generateUUID(),
  kinId: kin.id,
  summary: summary,
  messagesUpToId: lastCompactedMessage.id,
  isActive: true,
  createdAt: Date.now(),
})

// Deactivate the old snapshot
await db.update(compactingSnapshots)
  .set({ isActive: false })
  .where(and(
    eq(compactingSnapshots.kinId, kin.id),
    eq(compactingSnapshots.isActive, true),
    ne(compactingSnapshots.id, newSnapshot.id),
  ))
```

#### Step 5 — Clean up old snapshots

If the number of snapshots exceeds `compacting.maxSnapshotsPerKin` (default: 10), the oldest ones are deleted.

## Catch-up loop

When a Kin has accumulated a large backlog of turns (e.g. 187 turns with a threshold of 25), a single compaction batch would only reduce the count by ~10 turns. To handle this, `maybeCompact()` runs in a **loop**: it keeps triggering compaction batches until the turn count drops below the threshold, up to a maximum of 20 cycles. SSE progress events include the current cycle and estimated total cycles so the UI can show meaningful progress.

## Memory extraction

Compacting also triggers the **memory extraction pipeline** on the messages that were just summarized.

### Extraction prompt

```
System: You are an assistant specialized in information extraction.
Analyze the exchanges below and extract information worth remembering long-term.

For each extracted piece of information, return a JSON object with:
- "content": the fact or knowledge (a clear, standalone sentence)
- "category": "fact" | "preference" | "decision" | "knowledge"
- "subject": the person or context concerned (name or "general")

Rules:
- Only extract **durable** information (not ephemeral details)
- Check if the information contradicts or updates an existing memory
- Do not duplicate information already present in existing memories

## Existing memories (to avoid duplicates)

{existing_memories_summary}

## Exchanges to analyze

{formatted_messages}

Return a JSON array. If nothing new to remember, return [].
```

### Processing results

```typescript
const extracted = await llm.chat({
  model: config.memory.extractionModel ?? 'claude-haiku-4-5-20251001',
  messages: [{ role: 'system', content: extractionPrompt }],
  // Force JSON output
})

for (const memory of extracted) {
  // Generate embedding
  const embedding = await embeddingProvider.embed({ input: memory.content })

  // Check for duplicates via semantic similarity
  const similar = await searchMemoriesByVector(kin.id, embedding, threshold: 0.95)

  if (similar.length > 0) {
    // Update existing memory if content has changed
    await updateMemory(similar[0].id, memory.content)
  } else {
    // New memory
    await createMemory({
      kinId: kin.id,
      content: memory.content,
      embedding: embedding,
      category: memory.category,
      subject: memory.subject,
      sourceMessageId: lastCompactedMessage.id,
      sourceChannel: 'automatic',
    })
  }
}
```

## Purge and rollback

### Purge

Removes the active snapshot. The Kin starts fresh at the next exchange (all non-compacted messages will be in raw context, until the next compacting cycle).

```typescript
await db.update(compactingSnapshots)
  .set({ isActive: false })
  .where(and(
    eq(compactingSnapshots.kinId, kinId),
    eq(compactingSnapshots.isActive, true),
  ))
```

> Original messages are never deleted.

### Rollback

Reactivates a previous snapshot. All messages between the old snapshot and the current snapshot will be re-compacted at the next cycle.

```typescript
// Deactivate the current snapshot
await db.update(compactingSnapshots)
  .set({ isActive: false })
  .where(and(
    eq(compactingSnapshots.kinId, kinId),
    eq(compactingSnapshots.isActive, true),
  ))

// Reactivate the target snapshot
await db.update(compactingSnapshots)
  .set({ isActive: true })
  .where(eq(compactingSnapshots.id, targetSnapshotId))
```

## SSE events

| Event | Description |
|---|---|
| `compacting:start` | Compaction started for a Kin (includes cycle number and estimated total for catch-up progress) |
| `compacting:done` | Compaction completed (includes summary and memories extracted) |
| `compacting:error` | Compaction failed (prevents infinite spinner in the UI) |
